# Architecture

## Overview

Single-node Debian homelab server configured with Ansible roles. The same server runs base services and a K3s cluster for containerised workloads.

## Role Structure

Each role owns exactly one concern. Roles never configure each other's resources directly — shared state goes through `inventory/group_vars/`.

```
scripts/init-workstation.sh → (run once on workstation) creates ~/.homelab-secrets/ tree, generates all three keys, writes SSH alias
bootstrap_user    → (bootstrap only) creates ansible OS user, installs public key, grants passwordless sudo
security          → SSH hardening, fail2ban, unattended-upgrades
firewall          → ufw rules (port list sourced from inventory/group_vars/homelab/vars.yml)
cockpit           → web management UI
storage           → shared, app-agnostic host directory roots for K3s apps (media/config/cache)
samba             → guest, read-write SMB share exposing /srv/media over the network
k3s               → K3s platform installation and configuration (wait for node Ready)
flux_auth         → Flux CD deploy-key lifecycle (verify key exists, display pubkey, gate before bootstrap)
flux_bootstrap    → Flux CD bootstrap (install flux CLI, run flux bootstrap git, inject sops-age Secret)
```

## Fresh Clone Readiness

Basic host bootstrap requires local Ansible tooling, Galaxy collections, edited inventories, and a
Debian server with a password-login sudo admin. It does **not** require SOPS, Cloudflare, or any
existing key material; `scripts/init-workstation.sh` generates all three keys (ansible SSH, flux-deploy SSH,
SOPS age) before the bootstrap playbook runs.

GitOps secrets are a later stage. The SOPS Age key (`~/.homelab-secrets/age/homelab.agekey`), the
`age1…` recipient in `.sops.yaml` in the `homelab-cluster` repo, and the Cloudflare token encryption
must be ready before encrypted cluster secrets or cert-manager DNS-01 are expected to reconcile.

## Playbook Execution Order

`playbooks/site.yml` applies roles in this order, ensuring dependencies are met:

1. `security` — SSH hardened before firewall locks down
2. `firewall` — applies port list from `inventory/group_vars/homelab/vars.yml`
3. `cockpit` — web UI (port already open by firewall)
4. `storage` — ensures shared directory roots exist (`/srv/media`, `/srv/config`, `/srv/cache`, ...), owned `1000:1000`, before any app could reference them
5. `samba` — exposes `/srv/media` as a guest, read-write SMB share (force user/group `1000:1000` so guest writes match the directory's existing ownership); depends on `storage` having created the directory
6. `k3s` — installs K3s, waits for node Ready
7. `flux_auth` — on first run: asserts deploy key exists (generated by `scripts/init-workstation.sh` before bootstrap), displays public key, pauses for GitHub registration, verifies SSH connectivity
8. `flux_bootstrap` — installs flux CLI, runs `flux bootstrap git`, then injects the SOPS age key into `flux-system/sops-age` (always; fails fast if the Age key is missing)

The `ansible` automation user is provisioned separately via `playbooks/bootstrap-user.yml` (see Access model below) — not as part of `site.yml`. `playbooks/teardown-k3s.yml` is a separate, destructive playbook that deletes K3s (service, data, certs) so `site.yml` can reprovision it from scratch; it refuses to run without `-e teardown_confirm=true`. `scripts/cluster.sh` wraps bootstrap/teardown/verify of the K3s+Flux cluster in an interactive menu, and `scripts/quick-deploy.sh` chains `bootstrap-user.yml` → `site.yml` → `verify.yml` for a fresh server in one run.

## Access model

You create a human **admin** (password login + sudo) during the Debian install. Before touching the server:

1. Run `scripts/init-workstation.sh` on the **workstation**: creates `~/.homelab-secrets/`, generates all
   three keys (ansible SSH, flux-deploy SSH, SOPS age), and writes the SSH alias.

Then run `playbooks/bootstrap-user.yml` with `inventory/bootstrap.yml`, connecting as the admin over
password (`--ask-pass --ask-become-pass`). The bootstrap playbook runs `bootstrap_user` on the **VPS**:
creates the **`ansible`** user (sudo group, `/bin/bash`), installs the ansible public key, and writes
`/etc/sudoers.d/ansible` (`NOPASSWD:ALL`).

Every run after uses `playbooks/site.yml`, connecting as `ansible` by that key — no `--ask-pass`/`--ask-become-pass`. The `security` role forces the `ansible` user to key-only (`PasswordAuthentication no` inside a `Match User ansible` block); the admin keeps password SSH as a break-glass login. Run `security` **after** confirming `ansible` key login, so a misconfigured key can't lock automation out.

## K3s Application Pattern

K3s apps do not belong in this repo's Ansible tree. Ansible installs the K3s platform and bootstraps
Flux CD once. From then on:

- **Flux CD** runs inside K3s and watches the **[homelab-cluster](https://github.com/rene-ott/homelab-cluster)** repo on GitHub
- Flux applies the HelmRelease CRDs and K8s manifests from there automatically
- An app change = a commit in `homelab-cluster` under `apps/` → Flux syncs the cluster

Ansible does not touch K8s apps after the Flux CD bootstrap.

### Flux bootstrap (token-less, deploy key)

`flux_auth` and `flux_bootstrap` together bootstrap Flux with `flux bootstrap git` over SSH
(`--path=clusters/core`, targeting the `homelab-cluster` repo) using a repo-scoped **deploy key**
— not a GitHub PAT. The flow:

1. `scripts/init-workstation.sh` (run before bootstrap) generates a passphrase-less ed25519 deploy keypair at
   `~/.homelab-secrets/ssh/flux-deploy` — idempotent (generate-if-missing), never overwrites.
   `flux_auth` (in `site.yml`) asserts the key exists, prints the public key, and pauses for the
   operator to register it on GitHub.
2. You add that public key as a repo **deploy key with write access** (GitHub > Settings > Deploy
   keys). Write is required so the first bootstrap can push the `gotk-*` manifests; post-bootstrap
   reconciliation is read-only (write may be revoked unless image automation is planned).
3. `flux_bootstrap` stages the private key in a `0600` temp file on the server, runs `flux bootstrap
   git`, and deletes the temp file (the key persists only in the in-cluster `flux-system` Secret).

This is provider-agnostic (any git host over SSH) and avoids a user-scoped, expiring PAT. See
`CLAUDE.md` rule #9 for the deploy-key carve-out.

**If you forked `homelab-cluster`:** delete `clusters/core/flux-system/` in that repo before the
first bootstrap so Flux regenerates it for *your* repo URL. (If you keep it, bootstrap rewrites the
hardcoded `gotk-sync.yaml` URL for you — but that push, like the fresh-seed push, needs the deploy
key to have **write** access.)

### Secrets into the cluster (Flux SOPS)

Kubernetes runtime secrets are committed to the `homelab-cluster` GitOps repo as SOPS-encrypted
manifests and decrypted in-cluster by Flux's kustomize-controller. This keeps the GitOps tree fully
declarative and reproducible without plain-text secrets in git.

**Secret flow:**
1. Secrets are encrypted with `sops` (using the homelab age key at `~/.homelab-secrets/age/homelab.agekey`)
   and committed as `*.sops.yaml` files in `homelab-cluster`.
2. `.sops.yaml` at the root of `homelab-cluster` configures the SOPS creation rule
   (`encrypted_regex: ^(data|stringData)$`).
   **Before encrypting any manifest, replace the `AGE_PUBLIC_KEY_HERE` placeholder in `.sops.yaml`
   (in `homelab-cluster`) with the real `age1…` public key — readable from `~/.homelab-secrets/age/homelab.agekey`.**
3. The `flux_bootstrap` role injects the age private key into the cluster as Secret
   `flux-system/sops-age` (key `age.agekey`, `no_log: true`) after every Flux bootstrap.
   Ansible fails fast with a clear remediation message if the Age key is absent.
4. The `cert-manager-configs` Flux Kustomization has `decryption: { provider: sops, secretRef: sops-age }`,
   so kustomize-controller decrypts all `*.sops.yaml` manifests under `infrastructure/configs/` in `homelab-cluster`.

The Cloudflare token lives as Secret `cloudflare-api-token` in namespace `cert-manager`
(`infrastructure/configs/cert-manager/overlays/core/cloudflare-api-token.sops.yaml` in `homelab-cluster`)
and must be encrypted with SOPS before a real token is committed.

To re-inject the age key (e.g. after a cluster wipe):
```bash
ansible-playbook playbooks/site.yml --tags flux_bootstrap
```

**Secret ownership model (decided 2026-06-24, Option A):** Ansible owns the two bootstrap secrets
(Flux deploy key, SOPS age key) as workstation files — not as inventory vars. The Flux deploy key is
generated by `scripts/init-workstation.sh`; the age key is read from `~/.homelab-secrets/age/homelab.agekey`
(`flux_bootstrap`). Kubernetes runtime secrets (Cloudflare token, etc.) belong in the `homelab-cluster`
GitOps repo, not in Ansible inventory. `inventory/group_vars/homelab/secrets.sops.yml` does not
exist and the `community.sops` vars plugin is not enabled. Reconsider inventory SOPS only when a
real per-host Ansible secret (not a workstation key or cluster runtime secret) appears.

### Local workstation files

The inventories key the host as `atlas`, which matches the SSH `Host atlas` alias — no
`ansible_host` override is needed. The real server address is never committed —
`scripts/init-workstation.sh` prompts for it and writes it into `~/.homelab-secrets/ssh/config`, which is included
in `~/.ssh/config`. All homelab secrets and config live under a single directory:

```text
~/.homelab-secrets/
├── ssh/
│   ├── ansible          # private key for Ansible SSH login
│   ├── ansible.pub
│   ├── flux-deploy      # private key for Flux GitHub deploy-key bootstrap
│   ├── flux-deploy.pub
│   └── config           # SSH Host atlas alias (HostName, User, IdentityFile)
└── age/
    └── homelab.agekey   # SOPS age keypair — private key stays local, never committed

~/.homelab-backups/
├── secrets/             # encrypted secrets backups (scripts/backup-secrets.sh)
└── config/              # plain /srv/config backups (scripts/backup-config.sh)
```

`~/.ssh/config` contains one line: `Include ~/.homelab-secrets/ssh/config`.

### Local secrets backup and restore

`scripts/backup-secrets.sh` is a local disaster-recovery helper. It is **not** part of Ansible `site.yml` and is never run automatically.

**What it backs up:**
- `~/.homelab-secrets/ssh/ansible` and `.pub`
- `~/.homelab-secrets/ssh/flux-deploy` and `.pub`
- `~/.homelab-secrets/ssh/config` (the atlas address alias)
- `~/.homelab-secrets/age/homelab.agekey`

**What it does not back up:** the full `~/.ssh/config` (it may contain unrelated hosts — only the homelab-specific include file is backed up), cluster secrets (those live as `*.sops.yaml` in `homelab-cluster`, encrypted and committed to git), GitHub deploy-key registration (re-run `flux_auth`), or any server-side state.

**Encryption:** Age passphrase mode (`age -p`). The backup is not encrypted with the homelab Age key — that key is *inside* the backup, so using it would be circular. The passphrase must be kept in a password manager.

**Backup location:** `~/.homelab-backups/secrets/YYYYMMDD-HHMMSS.tar.gz.age`

```bash
# Interactive — prompts for backup or restore, then restore source
scripts/backup-secrets.sh
```

`restore` prompts before overwriting any conflicting file and also ensures `~/.ssh/config` contains
`Include ~/.homelab-secrets/ssh/config`, inserting at the top if missing. Auto-detection fails if
zero or more than one backup is present — delete extras manually.

### Server config backup and restore

`scripts/backup-config.sh` is the server-side counterpart to `backup-secrets.sh`, same interactive
backup/restore menu style. It backs up and restores `/srv/config` (the host directory K3s apps
write their config into via hostPath) between the server and the workstation. It is **not** part of
Ansible `site.yml` and is never run automatically.

**Backup:** streams the whole `/srv/config` tree over SSH (`ssh atlas sudo tar czf - -C /srv
config`) straight into a timestamped tarball — no temp file is ever left on the server. This is a
first pass: the archive is a plain, **unencrypted** tarball, and there is no retention/pruning of
old backups.

**Backup location:** `~/.homelab-backups/config/config-YYYYMMDD-HHMMSS.tar.gz`

**Restore:** lists all local backups (newest first), prompts which one to extract (default: most
recent), confirms before extracting, then streams it back over SSH (`ssh atlas sudo tar xzf - -C
/srv --numeric-owner`). Files already on the server with the same path are overwritten; anything
else under `/srv/config` is left untouched (extraction never deletes).

```bash
scripts/backup-config.sh   # interactive — prompts for backup or restore
```

## Port Management

All host-level firewall ports are declared in **`inventory/group_vars/homelab/vars.yml`** only. The `firewall` role is the sole actor that opens ports — no other role may call `ufw`, `nftables`, or `iptables` directly.

To add a port: edit `inventory/group_vars/homelab/vars.yml`, then run `ansible-playbook playbooks/site.yml --tags firewall`.

### Port list

| Port | Proto | Service | Sprint |
|------|-------|---------|--------|
| 22 | TCP | SSH | 004 ✓ |
| 9090 | TCP | Cockpit | 005 ✓ |
| 6443 | TCP | K3s API server | 008 ✓ |
| 80 | TCP | HTTP ingress (Traefik) | 008 ✓ |
| 443 | TCP | HTTPS ingress (Traefik) | 008 ✓ |
| 445 | TCP | Samba/SMB (media share, guest read-write) | — |

## Variable Files

| File | Scope |
|------|-------|
| `inventory/group_vars/all.yml` | Bootstrap/access vars — `ansible_admin_user`, `ansible_admin_key_path`, `ansible_admin_sudo_group`, `homelab_local_ssh_key_dir`; applies to all inventories |
| `inventory/group_vars/homelab/vars.yml` | Non-secret operational vars for the `homelab` group: SSH port, apt cache, firewall ports, K3s/Flux versions + bootstrap coordinates |
| `inventory/group_vars/homelab/secrets.sops.yml.example` | Documentation only — describes a future optional encrypted-secrets path; intentionally inactive (see secret ownership model above) |

### Where each value lives

All non-secret config is committed directly to inventory or role defaults. There is no `.env` file
and no `lookup('env', ...)` in the codebase.

| Kind of value | Home |
|---------------|------|
| Per-host connection (`ansible_host`, `ansible_user`) | `inventory/hosts.yml` / `inventory/bootstrap.yml` (inline, committed) |
| Connection topology — `homelab` group | `inventory/hosts.yml` |
| Local SSH key directory | `inventory/group_vars/all.yml` (`homelab_local_ssh_key_dir`) |
| Firewall ports, SSH port, apt cache | `inventory/group_vars/homelab/vars.yml` |
| K3s version pin | `inventory/group_vars/homelab/vars.yml` |
| Flux version pin | `inventory/group_vars/homelab/vars.yml` |
| Flux bootstrap coordinates (URL, path) | `inventory/group_vars/homelab/vars.yml` |
| Workstation key paths (ansible key, Flux deploy key, age key) | `inventory/group_vars/all.yml` + `inventory/group_vars/homelab/vars.yml` |
| The one Ansible secret — SOPS age key | `~/.homelab-secrets/age/homelab.agekey` (workstation only, never committed) |
| Kubernetes runtime secrets (e.g. Cloudflare token) | `homelab-cluster` repo — `*.sops.yaml` (committed, SOPS-encrypted) |

The bootstrap admin isn't stored — pass it once on the first run via `-u <admin>`.

## Design Decisions

**One role per concern.** Roles are developed and tested independently; `site.yml` stays a thin orchestrator, and new services are added as new roles without touching existing ones.

**Bootstrap is a separate playbook.** `playbooks/bootstrap-user.yml` with `inventory/bootstrap.yml` handles the one-time (or recovery) provisioning of the `ansible` OS user over password. `playbooks/site.yml` only ever connects as `ansible` by key — the two flows never mix.

**Firewall ports in group_vars only.** `inventory/group_vars/homelab/vars.yml` is the single source of truth for open ports; service roles don't reason about firewall state, and a port change needs only a `--tags firewall` run.

**K3s platform is infrastructure, apps are GitOps.** This repo installs K3s and bootstraps Flux CD once; after that Flux manages all app deployments by watching the `homelab-cluster` repo — no Ansible in the app lifecycle.

## Verification

Deployment (`site.yml`) and verification (`playbooks/verify.yml`) are kept separate. `verify.yml` is
**read-only** — it asserts services are genuinely healthy (Cockpit serves a 200, the K3s node is
`Ready`, Flux is reconciling) without mutating anything, so a healthy run reports `changed=0`. Each
role owns its checks in `roles/<role>/tasks/verify.yml`, pulled in tagged by role (`--tags cockpit` /
`--tags k3s` / `--tags flux_bootstrap`); it runs **on demand**, and the per-role files are reusable
by a future Molecule `verify` step. Extend it by adding a `tasks/verify.yml` to a role as it gains
a running service.
